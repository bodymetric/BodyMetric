# Implementation Plan: Complete Missing User Profile Data

**Branch**: `005-profile-completion-form` | **Date**: 2026-04-12 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/005-profile-completion-form/spec.md`

## Summary

After a successful Google Sign-In, the token exchange response already contains the authenticated user's profile. This plan detects profile incompleteness (missing `name`, `height`, or `weight`) at that moment — zero extra network calls — and routes the user to a real `UpdateProfileView` form (replacing the current placeholder stub in `CreateUserView`). On successful `POST /api/users` (HTTP 201), the profile is persisted to `ProfileStore` and the user is routed to `HomeView`. The gate is re-applied on app restart if the profile is still incomplete.

## Key Design Decisions

### 1. Completeness check at auth time

`AuthService.signInWithGoogle()` parses `AuthUser` from `TokenExchangeResponse.user` after a successful exchange. If `name`, `height > 0`, `weight > 0` are all present → `ProfileStore.save(from:)` + `needsProfileSetup = false`. Otherwise → `needsProfileSetup = true` (profile not saved yet).

### 2. AuthUser — separate model from UserProfile

`AuthUser` decodes the `user` object in the auth response (`name`, `email`, `height`, `weight` — no unit strings). `UserProfile` continues to decode the `GET /api/users` response (`weight`, `weightUnit`, `height`, `heightUnit`). Separate types prevent field-conflation bugs.

### 3. ProfileStore.isComplete updated

New gate: `name` non-empty + `height > 0` + `weight > 0`. Unit strings removed from the gate (they are supplemental display data). `ProfileStore` gains a `name` field stored under `bm.profile.name`.

### 4. Navigation ownership in BodyMetricApp

`BodyMetricApp.authenticatedContainer` checks `authService.needsProfileSetup`. If true → `CreateUserView` (the real form). If false → `HomeView`. This keeps routing at the outermost layer and out of `HomeViewModel`.

### 5. UpdateProfileService goes through NetworkClient

`POST /api/users` uses `NetworkClientProtocol` just like `UserProfileService`. Bearer injection and 401 handling are free.

### 6. 4-second success redirect

`UpdateProfileViewModel` uses `Task.sleep(for: .seconds(4))` after showing the success message, then sets `navigationState = .home`. The `View` observes this and triggers the navigation. For tests, `redirectDelay` is injectable (default 4.0, set to 0.05 in tests).

### 7. Session restore path

`AuthService.restorePreviousSignIn()` checks `ProfileStore.isComplete` after restoring the Google session. If incomplete → `needsProfileSetup = true`. This ensures the gate is re-applied on cold launch.
